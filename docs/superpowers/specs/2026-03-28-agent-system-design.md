# Agent System & Prompt Architecture — Design Spec

## Goal

Build a multi-agent system for the Talkstore project where each agent has a defined role, a reusable profile (system prompt), and operates within a structured task lifecycle. The prepare phase (explorer + researcher) produces a per-task prompt file that all downstream agents (builder, reviewer, security, validator, compliance) consume. All agents use Claude Opus 4.6.

## Architecture Overview

Two types of files:

1. **Agent profiles** (`docs/agents/*.md` in the Talkstore project) — reusable system prompts that define how each agent behaves. Static across tasks, evolve with the project.
2. **Task prompt files** (`docs/prompts/task-{id}-{slug}.md` in the Talkstore project) — per-task documents generated by the prepare phase. Contain findings, decisions, build instructions, and criteria for all downstream agents.

The tracker stores a pointer to the prompt file: `builder_prompt: "docs/prompts/task-009-scaffold-shopify-remix.md"`. The MCP server's `get_task_context` reads and includes the prompt file contents when the field is set.

## Agent Roster

| Agent | Profile File | When It Runs | Purpose |
|-------|-------------|-------------|---------|
| Talkstore PM | `talkstore-pm.md` | Every session (silent) | Umbrella identity. Knows project state, timeline, priorities. Stays silent until asked. Orients via `get_project_status()` and `list_tasks()` at session start. |
| Explorer | `explorer.md` | Prepare phase | Read-only codebase investigation. Scans files, traces data flows, maps architecture, finds relevant patterns and dependencies. |
| Researcher | `researcher.md` | Prepare phase | External research. Looks up docs, best practices, known gotchas. Evaluates whether plugins/skills/MCPs could improve task execution. Asks operator clarifying questions when task has ambiguity. |
| Builder | `builder.md` | Start phase | Implementation. Reads the enriched prompt file, writes code, runs tests. Follows patterns from CLAUDE.md and codebase conventions. |
| Reviewer | `reviewer.md` | Post-build (automatic) | Code quality check. Reads the diff against acceptance criteria, checks for missed requirements, style issues, logic errors, convention violations. |
| Security | `security.md` | Post-build (automatic) | Security scan. Checks for vulnerabilities, Shopify-specific security issues (session tokens, HMAC validation, secrets exposure, XSS, injection). |
| Validator | `validator.md` | Post-build (automatic) | Runs `npm run build`, `npm run typecheck`, `npm run lint`, and any task-specific test commands. Verifies everything passes. |
| Compliance | `compliance.md` | Post-build (automatic) | Checks which submission checklist items this task affects (if any). Calls `get_checklist_status()` to see current state. Verifies affected items are actually satisfied by the code. Skips gracefully when no items are relevant. |

All agents use **Claude Opus 4.6**.

## Task Lifecycle with Agents

### Prepare Phase

Operator says: "Prepare task 009"

1. Agent loads `talkstore-pm.md` profile (already loaded at session start)
2. Agent loads `explorer.md` profile, calls `get_task_context(task_id)`, investigates codebase
3. Agent loads `researcher.md` profile, does web research, evaluates plugins/skills/MCPs, asks operator questions if task has ambiguity
4. Agent writes the prompt file to `docs/prompts/task-009-scaffold-shopify-remix.md`
5. Agent updates tracker: sets `builder_prompt` field to the file path via MCP
6. Task stays in **Todo** — no status change

### Start Phase

Operator says: "Start task 009"

7. Agent calls `start_task(task_id)` — task moves to **In Progress**
8. Agent loads `builder.md` profile, reads the prompt file
9. Agent implements the task — writes code, runs tests
10. Builder finishes — automatic post-build chain begins:

### Post-Build Chain (Automatic)

11. Agent loads `reviewer.md` profile, reviews the code against acceptance criteria and review criteria from prompt file. Logs findings via `log_action`. Fixes any issues found.
12. Agent loads `security.md` profile, scans for vulnerabilities and Shopify security issues. Logs findings via `log_action`. Fixes any issues found.
13. Agent loads `validator.md` profile, runs build/typecheck/lint/tests. Logs results via `log_action`. Fixes any failures.
14. Agent loads `compliance.md` profile, checks if any submission checklist items are affected. Calls `get_checklist_status()`. If items are affected, verifies they're satisfied. Logs via `log_action`. Calls `toggle_checklist_item` if newly satisfied.
15. If any agent found issues that required fixes, re-run the chain from step 11 to verify fixes didn't break anything.
16. All four pass cleanly — agent calls `complete_task(task_id, summary)` — task moves to **Review**

### Feedback Loop

Operator reviews in the Review column.

- **Actionable feedback** ("fix X", "change Y") — agent calls `reject_task`, loads builder profile, fixes, re-runs post-build chain, calls `complete_task`
- **Conversational feedback** ("what about X?", "research Y") — agent answers/researches, task stays in Review
- **Conversational then actionable** — agent answers first, then when operator says to take action, calls `reject_task`, fixes, re-runs chain

### Completion

Operator says: "Complete task 009" — agent calls `approve_task(task_id)` — task moves to **Done**.

## Per-Task Prompt File Format

Generated by the prepare phase, stored at `docs/prompts/task-{id}-{slug}.md`:

```markdown
# Task {id}: {label}

## Overview
What this task is, why it matters, how it fits in the project timeline.

## Codebase Findings
Explorer's findings — existing files, patterns, dependencies, relevant code.

## Research Findings
Researcher's findings — docs, best practices, external references.
Any recommended plugins/skills/MCPs for this task.

## Your Decisions
Answers to questions the agent asked the operator during prepare.

## Build Instructions
Detailed implementation brief for the builder — what to build, how,
what patterns to follow, what files to create/modify.

## Acceptance Criteria
- [ ] Criterion 1
- [ ] Criterion 2

## Constraints
- Do not X
- Do not Y

## Context Files
- `path/to/file.ts` — why it matters

## Reference Docs
- https://docs.example.com — what it covers

## Review Criteria
What the reviewer should check beyond acceptance criteria.

## Security Checks
Task-specific security concerns for the security agent.

## Validation Steps
Specific commands to run, what passing looks like.

## Compliance
Which submission checklist items this task affects. "None" if not applicable.
```

## MCP Server Changes Required

### New tool: `enrich_task`

The prepare phase needs to write enrichment fields back to the tracker. The current `update_task` only supports `priority`, `assignee`, `execution_mode`, and `notes`. Add a new tool:

```
enrich_task(task_id, {
  prompt?,                  // prompt override text
  builder_prompt?,          // path to the prompt file
  acceptance_criteria?,     // string array
  constraints?,             // string array
  context_files?,           // string array
  reference_docs?           // string array
})
```

Only provided fields are updated. Logs to agent_log.

### Update `get_task_context`

When `builder_prompt` is set on a subtask, `get_task_context` reads the file at that path (relative to TALKSTORE_ROOT) and includes its contents in the context output under a `# Task Prompt` section.

### New tracker field: `builder_prompt`

Add `builder_prompt: string | null` to the `Subtask` interface in `mcp-server/src/tracker.ts`. This stores the relative path to the task's prompt file.

## Agent Profile Structure

Each profile in `docs/agents/*.md` follows this structure:

```markdown
# {Agent Name}

## Role
One-line description of what this agent does.

## Model
Claude Opus 4.6

## When You Run
When in the task lifecycle this agent is activated.

## Instructions
Detailed behavioral instructions for this agent role.

## Tools You Use
Which MCP tools this agent should call and when.

## Output
What this agent produces — findings, code changes, log entries, etc.
```

## Timeline-Aware Behavior

All agent profiles reference project timeline via `get_project_status()`:

- **Early project (weeks 1-6):** Prioritize architecture quality, thorough exploration, ask more questions. Foundations matter.
- **Mid project (weeks 7-14):** Balance speed with quality. Flag scope creep. Watch drift.
- **Late project (weeks 15-18):** Prioritize shipping. Minimize risk. No speculative refactors. Flag anything that could delay submission.
- **Final buffer (weeks 19-20):** Review-only. Bug fixes. Submission checklist items. No new features.

These guidelines are included in `talkstore-pm.md` and referenced by other profiles.

## What We Are NOT Building

- No separate processes or subprocesses — all agents run in the same Claude Code session
- No headless `claude --print` execution — everything is interactive
- No new Electron UI — the Command Center dashboard is read-only, shows status via tracker
- No authentication layer on MCP tools — operator trust model
- No changes to the Electron app — all changes are in the MCP server and the Talkstore project docs
